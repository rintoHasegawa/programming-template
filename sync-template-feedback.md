# /sync-template 改善提案

programming-template リポジトリの `.claude/commands/sync-template.md` について，実運用で発覚した問題点と改善案をまとめる．

## 背景

MiniTIAS プロジェクト（Flutter 製）で初回 `/sync-template` を実行したところ，以下の問題が発生した:

- テンプレート側の最小限な `.gitignore`（コメントアウトされた placeholder のみ）が，`flutter create` が自動生成した Flutter 固有の `.gitignore` を丸ごと上書きした
- 結果，`.dart_tool/`，`build/`，`*.iml`，`.idea/` 等の除外ルールが失われ，ビルド成果物が Git の追跡対象になりかけた
- `CLAUDE.md` もテンプレート側の placeholder にほぼ全置換される挙動だった（こちらは sync 担当の AI が気付いて手動マージしたため事なきを得た）

現状の sync-template.md はステップ 5 で `cp` による一括上書きを行うだけで，プロジェクト固有の内容を保持する仕組みがない．注意事項（末尾）で「ファイルコピーによりプロジェクト固有の変更が上書きされる場合は `git diff` で確認してユーザーに報告する」と記載されているが，**コピー後に diff を見るのでは手遅れ**（特に `.gitignore` のような全書き換えされるファイル）．

## 問題点の整理

### 1. マージが必要なファイルの扱いが未定義

以下のファイルは本質的に「プロジェクト固有の内容 + テンプレートの共通ルール」のハイブリッドだが，現状は盲目的に上書きされる:

| ファイル | プロジェクト固有の内容 | テンプレートの共通ルール |
| --- | --- | --- |
| `.gitignore` | フレームワーク固有の除外ルール（Flutter/Node.js 等） | `.claude/commit-context.md` などテンプレート都合の除外 |
| `CLAUDE.md` | プロジェクト名・概要・開発進捗・固有規約 | エージェントチーム節・Git 運用節の共通記述 |
| `.gitattributes` | プロジェクトによって改行コード・マージ戦略が異なる | 改行コード統一（`* text=auto eol=lf`） |

### 2. 初回同期の挙動が危険

初回同期では `.claude/template-sync-sha` が存在しないため，テンプレートの**全ファイルを A（追加）扱い**で取り込む．しかし実際には:

- `.gitignore` は `flutter create` / `npm init` 等で既に生成されている
- `CLAUDE.md` も `/init` スラッシュコマンドで作られている可能性がある

つまり「A 扱い = 既存ファイルなし」という前提が崩れているのに，処理は上書き一択になっている．

### 3. 事後検証では遅い

ステップ 4 の変更一覧提示はファイル名の列挙のみ．中身の差分は表示されない．ユーザーは「取り込み開始します」と言われて OK するが，実際に何が失われるかはコピー後の `git diff` を見て初めて分かる．

## 改善案

### A. 「マージ必須ファイル」を明文化

sync-template.md に新セクションを追加:

```markdown
## マージ必須ファイル (Merge-required Files)

以下のファイルは**上書きせず**，テンプレートの変更内容を既存ファイルにマージする．
コピー前にチェックし，既存ファイルがある場合はマージ処理に分岐する．

| ファイル | 理由 | マージ方針 |
| --- | --- | --- |
| `.gitignore` | フレームワーク固有ルール（Flutter/Node 等）を保持する必要がある | テンプレート側で追加された行のみを既存ファイル末尾に追加し，既存行は触らない |
| `CLAUDE.md` | プロジェクト名・進捗・固有規約を保持する必要がある | テンプレートで変更された共通セクション（エージェントチーム・Git 運用等）のみをピンポイントで更新する．プロジェクト固有セクションは触らない |
| `.gitattributes` | プロジェクトによって設定が異なる可能性がある | 差分を表示してユーザーの判断に委ねる |
```

### B. ステップ 5 の処理を分岐させる

コピー前にマージ必須リストと照合する:

```bash
MERGE_FILES=(".gitignore" "CLAUDE.md" ".gitattributes")

is_merge_file() {
  local target="$1"
  for f in "${MERGE_FILES[@]}"; do
    [ "$f" = "$target" ] && return 0
  done
  return 1
}

echo "$CHANGED_ENTRIES" | while IFS=$'\t' read -r status file newfile; do
  [ -z "$status" ] && continue
  case "$status" in
    A|M)
      if is_merge_file "$file" && [ -f "$file" ]; then
        # マージ処理に分岐（下記 C を参照）
        handle_merge "$file" "$TEMP_DIR/$file"
      else
        mkdir -p "$(dirname "$file")"
        cp "$TEMP_DIR/$file" "$file"
      fi
      ;;
    R*)
      ...
      ;;
  esac
done
```

### C. マージ処理の具体的な手順

各マージ必須ファイルごとにマージ戦略を明文化する:

**`.gitignore` の場合**
1. 既存の `.gitignore` を読み込む
2. テンプレートの `.gitignore` と行単位で比較
3. テンプレートにしかない行を既存ファイル末尾に追記（Claude Code 関連の既知コメントセクションがあればそこに追記）
4. 結果を diff で表示して確認

**`CLAUDE.md` の場合**
1. 既存ファイルを読み込む
2. AI（同期担当エージェント）が以下を判断:
   - テンプレートで変更された共通セクション（エージェントチーム，Git 運用，ドキュメント構成等）を特定
   - プロジェクト固有セクション（プロジェクト名・概要，開発進捗，固有規約等）を保持
   - 該当セクションのみを Edit ツールで更新
3. 結果を diff で表示して確認

**`.gitattributes` の場合**
1. 既存ファイルとテンプレートの diff を表示
2. ユーザーに「上書き / マージ / スキップ」を問う

### D. ステップ 4 に差分プレビューを追加

変更一覧提示時に，マージ必須ファイルに変更がある場合は差分サマリーを先出しする:

```markdown
**⚠ マージが必要なファイル:**
- `.gitignore` （既存 48 行 → テンプレート 9 行．Flutter 固有ルール 38 行を保持する必要があります）
- `CLAUDE.md` （既存にプロジェクト固有セクションが 40 行あります．テンプレート更新箇所のみマージします）

取り込みを開始します．
```

これにより，ユーザーは OK する前に「何が保持され，何が更新されるか」を把握できる．

## 補足（優先度低）

### E. `main` / `master` ブランチ名の自動判定

sync-template.md 自体は PR 作成を行わないが，後続の `/commit push` で PR を作る際に base ブランチをハードコードしないよう，以下を利用する:

```bash
DEFAULT_BRANCH=$(gh repo view --json defaultBranchRef --jq '.defaultBranchRef.name')
```

MiniTIAS は `master` を既定ブランチにしているため，`main` 前提の挙動だとエラーになる．

### F. `.claude/plans/` 等の gitignored ディレクトリの扱い

テンプレート側に `.claude/plans/` を含めていないのは正しいが，プロジェクトによっては同名ディレクトリが既にあり得る．特別な配慮は不要だが，ドキュメントで「テンプレートは `.claude/` 配下のうち `agents/`，`commands/`，`hooks/`，`settings.json`，`template-sync-sha` のみを管理する」と明示するとよい．

## まとめ（テンプレートメンテナへの依頼）

実運用で明らかになった主な改善ポイントは以下の 2 点:

1. **`.gitignore` と `CLAUDE.md` を「マージ必須ファイル」として定義し，上書きではなく差分マージする処理をステップ 5 に組み込む**
2. **ステップ 4 でマージ対象ファイルの差分プレビューを先出しし，ユーザーが取り込み前に影響範囲を把握できるようにする**

特に 1 は未対応だと新規プロジェクトのセットアップ時に必ず踏む地雷になる（Flutter プロジェクトで実証済み）．初回同期を想定した対応を強く推奨する．
